# 최종 발표 스크립트 — nyam!

> 예상 소요 시간: 8~10분

---

## 슬라이드 1: 표지 (30초)

안녕하세요, 파이널 2팀 nyam! 프로젝트 최종 발표를 시작하겠습니다.

nyam!은 "Next Your Appropriate Menu"의 약자로, 사용자의 현재 컨디션에 딱 맞는 메뉴와 맛집을 추천해주는 서비스입니다. '냠'이라는 이름에는 음식을 먹는 소리의 직관성과, 결정 장애를 '냠'하고 먹어치워 해결해준다는 의미가 담겨 있습니다.

---

## 슬라이드 2: 문제 정의 & 해결 방향 (1분)

저희가 주목한 문제는 "오늘 뭐 먹지?"라는 매일 반복되는 결정 피로입니다.

사실 현대인에게 음식 정보가 부족한 것이 아닙니다. 오히려 선택지가 너무 많아서 비교 비용이 늘어나고, 결국 익숙한 메뉴로 돌아가거나 결정을 미루게 됩니다. 문제의 본질은 정보 부족이 아니라, 현재 컨디션을 기준으로 의사결정을 정리해주는 구조의 부재입니다.

nyam!은 이를 "No Text, Only Swipe" 방식으로 해결합니다. 텍스트 입력 없이 스와이프만으로 컨디션을 입력받아, 룰 기반으로 음식 타입을 추론하고, 근처 식당을 별점과 거리 기준으로 추천합니다.

---

## 슬라이드 3: 프로젝트 로드맵 (1분)

전체 프로젝트는 5주간 진행되었습니다.

**1주차 기획 단계**에서는 PRD를 v1.0과 v0.2 두 버전으로 작성하며 서비스의 방향을 잡았고, 페르소나 정의, 컨디션-메뉴 매핑 설계, 데이터 수집 전략을 수립했습니다.

**2주차 디자인 단계**에서는 6개 화면의 UX 명세를 정의하고, 스와이프와 채팅 두 가지 인터랙션을 설계했습니다. 로고와 심볼, 폰트도 직접 아이디어를 내서 제작했고, 프론트엔드 UI는 개발 편의성을 위해 shadcn/ui 라이브러리를 채택했습니다.

**3~4주차 개발 단계**에서는 프론트엔드와 백엔드를 병행 개발하며, v1.0부터 v1.5까지 총 6회의 반복 릴리즈를 진행했습니다.

**5주차**에는 최종 기능 추가와 발표 자료를 준비했습니다.

## 슬라이드 4: 기획 단계 (1분)

기획의 핵심은 **두 가지 추천 방식을 설계**한 것입니다.

첫째는 **AI 채팅 방식**으로, GPT-4o-mini가 자연어 대화를 통해 컨디션을 파악하고 메뉴와 맛집을 추천합니다. "피곤해요"라고 말하면 삼계탕이나 국밥을 추천하고, 위치를 말하면 근처 맛집까지 검색해줍니다.

둘째는 **스와이프 방식**입니다. 4개의 이진 질문 — 자극적인가, 따뜻한가, 가볍게인가, 국물이 필요한가 — 을 스와이프로 답하면 컨디션 벡터가 생성되고, 8개 Food Type 중 최적 결과를 룰 기반으로 산출합니다. 16가지 모든 응답 조합에서 단일 Food Type이 항상 결정되도록 동점 처리까지 설계했습니다.

추천 점수 공식에서는 초근접 식당에 가산점을 부여해서, 도보 3분 이내 식당이 별점 차이만으로 밀리지 않도록 했습니다.

## 슬라이드 5: 로고 & 브랜드 디자인 (1분)

이 슬라이드는 저희가 직접 아이디어를 내서 제작한 브랜드 디자인 과정입니다.

네이밍부터 시작했습니다. "냠"이라는 이름에는 세 가지 의미가 있습니다. 음식을 먹는 소리의 직관성, "Y/N" 글자 구조가 스와이프 Yes/No와 시각적으로 매칭되는 점, 그리고 결정 장애를 "냠" 하고 먹어치워 해결해준다는 중의적 의미입니다.

심볼 캐릭터는 입을 크게 벌린 노란 캐릭터로, "먹는다"는 행위를 직관적으로 표현했습니다. 귀엽고 명쾌한 해결사 컨셉에 맞게 디자인했습니다.

로고 폰트는 Unbounded Bold를 선택했는데, 둥글고 볼드한 형태가 "냠" 소리의 통통 튀는 느낌과 잘 어울립니다. 소문자 표기로 친근함을 강조하고, 느낌표에 레드 컬러를 적용해 에너지를 부여했습니다.

컬러 시스템은 프라이머리 옐로우로 식욕과 따뜻함을, 레드 액센트로 에너지를, 다크 네이비로 신뢰감을 표현했습니다.

---

## 슬라이드 6: UI 컴포넌트 설계 (40초)

UI 구현에서 핵심 결정은 shadcn/ui 라이브러리의 채택이었습니다. Radix UI 기반의 접근성 보장, Tailwind CSS와의 완벽한 호환, 복사-붙여넣기 방식의 커스터마이징 자유도 — 이 세 가지가 5주 단기 개발에 최적의 선택이었습니다.

Dialog, DropdownMenu, ScrollArea, Toast 등의 shadcn 컴포넌트로 핵심 UI를 빠르게 구축하고, 거기에 커스텀 스와이프 카드와 LazyImage를 추가해 서비스 특성에 맞는 UX를 완성했습니다.

---

## 슬라이드 7: 주요 화면 설계 (1분)

이 슬라이드는 실제 화면 캡쳐와 함께 기획-디자인-개발 의도를 설명합니다.

**메인 랜딩 화면**입니다. 기획에서는 첫 화면에서 서비스의 세 가지 핵심 가치 — AI 맞춤 추천, 내 주변 맛집, 테마 컬렉션을 즉시 전달하도록 설계했습니다. 디자인에서는 마스코트 심볼을 중앙에 배치해 브랜드 인지도를 높이고, 옐로우 CTA 버튼으로 시선을 유도했습니다. 개발에서는 shadcn Dialog 기반 온보딩 3단계 플로우를 구현했습니다.

**AI 채팅 화면**입니다. 기획에서는 시간대별 맞춤 인사말과 퀵리플라이로 입력 부담을 최소화했습니다. 디자인에서는 봇 말풍선과 퀵리플라이 칩 구조, 마스코트 프로필로 대화에 인격을 부여했습니다. 개발에서는 GPT-4o-mini 스트리밍 응답과 카카오 API 맛집 검색을 연동했습니다.

**메인 페이지 전체**는 CatchTable을 벤치마킹해 다양한 진입점을 제공하고, 카카오 실시간 데이터와 LazyImage 최적화를 적용했습니다.

## 슬라이드 8: 개발 진척 사항 (1분 30초)

개발은 v1.0부터 v1.5까지 총 6회 릴리즈를 진행했습니다.

**v1.0**에서 AI 챗봇과 맛집 검색의 기본 뼈대를 만들었고, **v1.1**에서 메인 페이지와 상세 페이지를 추가하며 nyam! 브랜딩을 적용했습니다. **v1.2**에서 카카오 소셜 로그인과 JWT 인증을 구현했고, **v1.3**에서 PWA 지원, 토스트 알림, 이미지 최적화, 즐겨찾기를 추가했습니다.

**v1.4**가 가장 큰 전환점이었는데, 프론트엔드에 하드코딩되어 있던 가짜 맛집 데이터를 전부 카카오 Local API 실시간 검색으로 교체했습니다. 챗봇에서 "홍대 파스타"처럼 직접 검색하는 기능도 이때 추가했습니다.

마지막 **v1.5**에서는 게이미피케이션 요소로 채팅 내 스와이프 추천과 메뉴 월드컵을 추가하고, 장소 검색, 반경 선택, 시간대별 맞춤 추천 기능을 넣었습니다.

## 슬라이드 9: 시스템 아키텍처 (40초)

전체 시스템은 3-Tier 구조입니다.

프론트엔드는 React + TypeScript + Vite 조합이고, UI는 shadcn/ui와 Tailwind CSS, 지도는 Kakao Maps SDK를 사용합니다. PWA를 지원해서 모바일에서 앱처럼 설치가 가능합니다.

백엔드는 FastAPI로, 챗봇 API, 맛집 검색 API, 인증 API 세 가지 주요 라우터로 구성됩니다.

외부 서비스로는 OpenAI GPT-4o-mini, 카카오 Local·Maps·OAuth API, Supabase PostgreSQL을 사용하고, Railway에 Docker로 배포하고 있습니다.

## 슬라이드 10: 핵심 기능 상세 (40초)

핵심 기능을 여섯 가지로 정리하면:

AI 챗봇이 컨디션에 맞는 메뉴를 추천하고, 게이미피케이션으로 스와이프 추천과 메뉴 월드컵을 제공합니다. 위치 기반 실시간 맛집 검색이 가능하고, 카카오 실데이터로 리뷰·영업시간·지도를 보여줍니다. 카카오 소셜 로그인과 즐겨찾기 기능이 있고, PWA로 앱 수준의 UX를 제공합니다.

## 슬라이드 11: 기술적 챌린지 (1분)

개발 과정에서 특히 기억에 남는 기술적 챌린지가 네 가지 있습니다.

첫째, **카카오 API의 데이터 한계**입니다. 공식 API는 가게명과 좌표만 주기 때문에, 비공식 API에서 리뷰와 영업시간을 가져오고 og:image로 이미지를 확보하는 방식으로 해결했습니다.

둘째, **카카오맵 SDK와 React의 DOM 충돌**입니다. SDK가 innerHTML을 통째로 교체해서 React 관리 노드가 파괴되는 문제를, 맵 컨테이너를 완전히 분리하는 방식으로 해결했습니다.

셋째, **OAuth 쿠키 문제**입니다. 302 리다이렉트 시 쿠키가 저장되지 않아서, auth code exchange 패턴을 도입했습니다.

넷째, **Service Worker 캐싱 충돌**로 인증과 API 요청이 가로채지는 문제를, denylist 설정으로 해결했습니다.

## Q&A 예상 질문

**Q: shadcn/ui를 선택한 이유가 뭔가요? MUI나 Ant Design과 비교하면?**
A: shadcn/ui는 컴포넌트 코드를 직접 프로젝트에 복사해서 사용하기 때문에, 다른 UI 라이브러리와 달리 의존성이 가볍고 커스터마이징이 자유롭습니다. Tailwind CSS와의 호환성도 뛰어나고, Radix UI 기반이라 접근성이 기본 보장됩니다. 5주라는 짧은 기간에 빠르게 일관된 UI를 만들기에 가장 적합했습니다.

**Q: 카카오 비공식 API 사용에 문제는 없나요?**
A: 카카오 플레이스 웹페이지의 공개 데이터를 읽는 수준이라 현재까지 문제없이 동작합니다. 장기적으로는 자체 리뷰 시스템이나 공식 API 확장을 고려하고 있습니다.

**Q: 룰 기반 추천의 한계와 향후 개선 방향은?**
A: 현재는 4개 이진 질문으로 8가지 Food Type을 결정하는 룰 기반입니다. 향후에는 사용자 선택 로그와 만족도 데이터를 축적해서 키워드 가중치를 동적으로 조정하거나, LLM을 활용한 자연어 기반 컨디션 파악으로 확장할 수 있습니다.

**Q: GPT-4o-mini 비용은 어느 정도인가요?**
A: 대화 1회당 약 1,400토큰을 사용하고, GPT-4o-mini 기준으로 MVP 단계에서는 월 1,000~2,000원 수준입니다. 사용자가 1,000명으로 늘어나도 월 35,000원 정도로 부담이 적습니다.

**Q: PWA와 네이티브 앱의 차이는?**
A: PWA는 별도 앱스토어 등록 없이 브라우저에서 홈 화면에 추가로 설치할 수 있고, 오프라인 캐싱과 푸시 알림도 지원합니다. 네이티브 앱 대비 개발 비용이 훨씬 낮으면서도 앱에 가까운 UX를 제공할 수 있었습니다.
